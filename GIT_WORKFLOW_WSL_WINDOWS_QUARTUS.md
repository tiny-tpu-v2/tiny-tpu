# Git Workflow For WSL + Windows Quartus

This is the recommended Git workflow for your setup:

- use `git` from WSL
- keep the active Quartus project on the Windows filesystem
- invoke the Windows Quartus tools from WSL
- do not mix WSL Git and Windows Git on the same checkout

## Recommended Layout

Use one shared project tree that both WSL and Windows can access:

- Windows path: `C:\fpga\de1soc_project`
- WSL path: `/mnt/c/fpga/de1soc_project`

This gives you:

- a normal Windows path for Quartus
- a normal Linux shell path for Git, scripting, and editing

## Why This Is The Best Practical Setup

Objectively, this is the best compromise for your environment:

- Git is generally cleaner and less frustrating in WSL than in Windows
- Quartus is more reliable when the project lives on a native Windows path
- older Windows tools often behave badly with `\\wsl$...` or UNC-style paths
- running Windows Quartus executables from WSL gives you one command-line workflow

So the split should be:

- WSL handles source control, scripting, and most editing
- Windows Quartus handles synthesis, fitting, assembly, timing, and programming

## Core Rule

Do not use multiple Git clients against the same checkout.

If the repo is managed with WSL `git`, keep it that way. Avoid using these against the same working tree:

- Git for Windows
- Windows IDE Git integrations
- GitHub Desktop on Windows

Mixing clients is the easiest way to create:

- line-ending churn
- duplicate credential prompts
- confusing file-mode changes
- index inconsistencies

## Daily Workflow

Work from WSL in the project directory:

```bash
cd /mnt/c/fpga/de1soc_project
git status
git switch -c feature/my-change
```

Edit files from WSL or a WSL-aware editor such as VS Code Remote WSL.

Then run Quartus from WSL against the same tree:

```bash
cd /mnt/c/fpga/de1soc_project/quartus
'/mnt/c/altera_lite/25.1std/quartus/bin64/quartus_sh.exe' --flow compile de1_soc_top
```

Review outputs, fix issues, and commit from WSL:

```bash
cd /mnt/c/fpga/de1soc_project
git add .
git commit -m "Update DE1-SoC top-level and constraints"
```

## Recommended Git Configuration

Set these in WSL for this workflow:

```bash
git config --global core.autocrlf input
git config --global core.fileMode false
```

What they do:

- `core.autocrlf input` prevents Windows-style line ending churn while still accepting CRLF on input
- `core.fileMode false` avoids noisy permission-bit changes on the Windows-mounted filesystem

## Line Endings

Be explicit about line endings so source files stay stable.

Recommended default:

- keep HDL, Tcl, and source files as LF
- allow Windows convenience files like `.bat` to use CRLF if needed

Add a `.gitattributes` file like this:

```gitattributes
* text=auto
*.sv text eol=lf
*.v text eol=lf
*.vh text eol=lf
*.vhd text eol=lf
*.vhdl text eol=lf
*.tcl text eol=lf
*.qsf text eol=lf
*.qpf text eol=lf
*.sdc text eol=lf
*.py text eol=lf
*.sh text eol=lf
*.bat text eol=crlf
```

## What To Commit

Commit:

- RTL source files
- testbenches
- hand-written Tcl scripts
- project `.qsf` and `.qpf` if you want reproducible Quartus setup
- constraints such as `.sdc`
- board-specific pin assignments you intentionally maintain
- software sources for Nios V, if used

Usually do not commit:

- `db/`
- `incremental_db/`
- `output_files/`
- generated simulation libraries
- large autogenerated logs and reports unless you have a specific reason

## Suggested Quartus `.gitignore`

Use a Quartus-focused ignore file:

```gitignore
# Quartus generated databases
db/
incremental_db/

# Build outputs
output_files/

# Common reports/logs
*.rpt
*.summary
*.smsg
*.qws
*.jdi
*.sld

# Programmer and temporary outputs
*.sof
*.pof
*.jam
*.jbc
*.cdf

# Simulation outputs
transcript
vsim.wlf
work/
simulation/modelsim/

# OS/editor noise
.DS_Store
Thumbs.db
```

You may choose to commit `.sof` or `.cdf` files for release artifacts, but they should not usually be treated as source.

## Best Practice For Branching

Keep hardware changes isolated and reviewable:

- one branch for RTL changes
- one branch for constraints or pin assignment changes
- one branch for tooling or build-script changes

This makes it easier to separate:

- functional design regressions
- timing regressions
- board bring-up mistakes

## Alternative Option

There is a cleaner but more complex alternative:

- keep the canonical Git repo inside WSL, such as `~/src/de1soc_project`
- sync or export a build copy to `C:\fpga\de1soc_project` for Quartus

This gives the best Linux-side Git performance, but it introduces an extra sync step and more chances for drift. Only use this if `/mnt/c` performance becomes a real issue.

## Recommended Final Policy

Use this as the default operating model:

- canonical working tree: `/mnt/c/fpga/de1soc_project`
- source control: WSL `git` only
- editing: WSL-native tools
- synthesis/programming: Windows Quartus executables called from WSL

That gives you the simplest workflow with the fewest moving parts while keeping both Git and Quartus in the environment where they behave best.
